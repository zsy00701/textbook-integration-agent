# 系统设计

## 1. 总体架构

### 部署拓扑
```
┌─────────────────────────────────────────────┐
│  浏览器(SPA)                                │
│  Vite 构建产物 → /index.html /assets/...    │
└──────────────┬──────────────────────────────┘
               │ HTTP REST
               ▼
┌─────────────────────────────────────────────┐
│  FastAPI 单进程(uvicorn)                   │
│  ├─ /api/*    业务路由(6 个模块)           │
│  └─ /         StaticFiles(挂前端 dist)     │
└──┬────────────┬──────────┬─────────────────┘
   │            │          │
   ▼            ▼          ▼
┌──────┐  ┌──────────┐  ┌────────────┐
│DeepSeek│ │BGE-zh    │  │ ChromaDB   │
│-V4 Pro │ │(本地 CPU) │  │持久化向量库 │
│(API)   │ │          │  │            │
└──────┘  └──────────┘  └────────────┘
   │
   │  外网调用(唯一外部依赖)
```

**关键决策**:前后端**同进程托管**,FastAPI 用 `StaticFiles(html=True)` 挂载 `src/frontend/dist`。优势:
- 部署一个进程即可,魔搭/Docker 都简单
- 同源,无 CORS 问题
- dev 时单独跑 `npm run dev`,Vite 配 proxy 转发 `/api` 到后端

## 2. 数据流(端到端)

详见 `docs/Agent架构说明.md` §c。这里给一个简化版:

```
[Upload] PDF/MD/TXT/DOCX
    │
    ▼  PyMuPDF 流式 / docx-pyhton-docx / 正则
[Parser] → data/parsed/{book_id}.json (TextbookDoc)
    │
    ▼  ThreadPool(4) × 章节,每章一次 LLM 调用
[KG Extractor] → data/graphs/{book_id}.json (BookGraph)
    │
    ▼  对所有 BookGraph 的 nodes
[Aligner] BGE 嵌入 + 余弦阈值 + LLM 边界裁决
    │
    ▼  对每个簇
[Decision] merge / keep,挑最完整定义胜出
    │
    ▼  超 30% 时按 confidence 升序删 keep
[Compressor]
    │
    ▼  边重映射 + 去重
[Master Graph] → data/integrated/master_graph.json
                  data/integrated/decisions.json + stats.json
    │
    │ 并行:
    ├─ 前端 ECharts 渲染(force/tree/sankey)
    │
    ▼  对所有 parsed/*.json 做 chunking
[RAG Indexer] 600+80 chunk → BGE 嵌入 → ChromaDB
                                       (同时构建 BM25 索引)
    │
    ▼  /api/rag/query
[Retriever] 向量 top-10 + BM25 top-10 → RRF top-5
    │
    ▼
[Generator] LLM(强约束 prompt 注入 top-5 chunks)
    │
    ▼
{answer, citations[], source_chunks[]}

[Chat 反馈]
    │
    ▼  LLM 解析意图 → patches[]
[Patch Apply] 修改 decisions.json + master_graph.json
    │
    ▼
{reply, decisions_changed[], history}
```

## 3. 各层技术选型与理由

| 层 | 选型 | 备选 | 理由 |
|---|---|---|---|
| 后端 | **FastAPI** + uvicorn | Flask, Django | 异步、Pydantic 集成、OpenAPI 自动生成、文件上传简单 |
| 前端 | **Vite + Vue 3 + TypeScript + Pinia** | React, Svelte | Vite HMR 快,Vue 模板对图表组件友好,TS 帮助前后端 schema 同步 |
| 图谱可视化 | **ECharts** (graph + tree + sankey) | D3, AntV G6, Cytoscape | 一个库覆盖三视图(P1 加分),官方 dark 主题省 CSS 工作 |
| LLM | **DeepSeek-V4 Pro** | OpenAI, 通义千问 | 中文好、便宜(~1/10 OpenAI 价)、稳定 JSON 模式 |
| Embedding | **BGE-small-zh-v1.5** | OpenAI Embedding | 中文 SOTA、本地免费、512 token 与 chunking 大小匹配 |
| 向量库 | **ChromaDB**(persistent) | FAISS, Qdrant | 单文件持久化、零运维、CRUD 接口完整 |
| PDF 解析 | **PyMuPDF (fitz)** | pdfplumber, pypdf | 速度最快、API 最完整(可拿 dict blocks 含字号) |
| Word | python-docx | textract | docx 唯一稳定方案 |
| 部署 | 魔搭创空间 + Docker | Vercel/Railway | 免费、对 Python ML 友好,允许 ChromaDB 持久化 |

## 6. 关键工程决策

### 6.1 持久化:JSON 文件 + ChromaDB
- **不用关系型数据库**:整合是大批量读写 + 单查询是"全图谱拉取",JSON 文件足够;省去 schema 迁移
- **ChromaDB 单独管向量**:与业务数据解耦,索引重建不影响 graphs/

### 6.2 异步:BackgroundTasks + 全局信号量
- **解析串行化**:PyMuPDF 多线程触发 SIGSEGV(实测验证),用 `_PARSE_LOCK` 保护
- **抽取并发**:章节级 ThreadPoolExecutor(4),被 LLM API 限速天然控制
- **整合/索引**:专属 Lock,避免重复触发

### 6.3 错误隔离
- 单章 LLM 失败 → 跳过该章,整本不阻塞
- LLM JSON 解析失败 → 截取首末花括号兜底
- 解析失败 → meta 状态标 `failed` + error 信息,前端显示

### 6.4 前端状态管理:Pinia 单 store
- 7 本教材规模下不需要 modular store
- 一个 `useAppStore` 包含 textbooks / currentGraph / decisions / ragStatus / chatHistory
- 所有 API 调用都在 store actions 中,组件只 dispatch

### 6.5 部署:同进程托管
- 魔搭创空间限制 1 个公开端口
- FastAPI `StaticFiles(html=True)` 完美承担 SPA 托管
- Docker 镜像不需要 Node(只在 build 阶段需要)

## 7. 性能与容量上限

| 维度 | 实测/估计 |
|---|---|
| 单 PDF 解析(437MB / 600 页) | ~30-60s |
| 单本 KG 抽取(20 章 × 4 并发) | ~3-5 分钟 |
| 跨教材整合(7 本,1500 节点) | ~30-60s(嵌入 + ≤80 LLM 验证) |
| RAG 索引重建(7 本) | ~3-5 分钟(嵌入瓶颈) |
| 单次 RAG 查询 | ~2-4 秒(含 LLM 生成) |
| 端到端(上传 → 整合 → 索引完成) | ~12-25 分钟(7 本) |
| 内存峰值 | ~2 GB(BGE 模型 ~500MB + ChromaDB ~300MB + PyMuPDF 缓冲 ~600MB) |
| 磁盘占用(不含原 PDF) | ~250MB(parsed/ ~150 + chroma/ ~80) |

## 8. 安全与隐私

1. **API key 在 .env**,gitignore;`.env.example` 占位
2. **CORS 全开**(`*`)是因为部署后前后端同源,且这是赛事系统,不涉及用户私有数据
3. **上传文件名 sanitize**:正则去除特殊字符,防路径穿越
4. **LLM 提示词注入风险**:对话面板的反馈通过 LLM JSON 模式输出 patches,patches 应用前做了 schema 严格校验
