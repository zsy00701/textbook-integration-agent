# 需求分析

## 1. 问题分解

赛题要求"用 AI 帮教师把 7 本教材变成不到 2.1 本的精华,而且变完之后教学效果不打折"。这是一个**信息压缩同时保证教学完整性**的双约束问题,我们将其分解为以下子问题:

### 1.1 子问题 A:多源教材的统一表征
**问题**:7 本教材格式不一(PDF 为主,可能含 MD/TXT/DOCX),每本结构(章节划分、定义习惯、术语)都不同。
**子约束**:必须可流式解析(组胚学单本 437MB),不能一次性加载内存。
**我们的方案**:
- 统一抽象 `TextbookDoc { textbook_id, chapters: [{title, content, page_start, page_end}] }`
- PDF 解析:PyMuPDF 逐页 `page.get_text("dict")`,字号 + 正则双策略识别章节,过滤页眉页脚
- MD/TXT/DOCX 各自适配,但落到同一 schema

### 1.2 子问题 B:知识点的粒度定义
**问题**:什么算一个"知识点"?粒度太粗 → 整合空间小;太细 → 噪声多、关系爆炸。
**我们的判定标准**:
- **是知识点**:命名概念(如"动作电位")、命名定理/公式(如"欧姆定律")、命名方法(如"巴氏消毒法")、命名现象(如"道恩斯综合征")、命名结构(如"线粒体")、命名过程(如"减数分裂")
- **不是知识点**:章节标题、图表编号、参考文献条目、纯举例性描述
- **粒度量化**:每章 15-25 个知识点为目标(LLM prompt 显式约束)

这也是我们让 LLM 输出结构化 `category` 字段的原因 —— 后续可按类别筛选。

### 1.3 子问题 C:重复判定标准
**问题**:不同教材如何识别"同一个知识点"?
**我们的判定标准(双重)**:
1. **语义相似度** ≥ 0.93(BGE 余弦)→ 直接判同一概念
2. **0.85 ≤ 语义相似度 < 0.93** → 调 LLM 二次验证(共有 200 字定义对比 + reason)
3. < 0.85 → 视为不同概念

**为什么不用纯阈值**:中文医学领域常出现"白细胞"vs"白细胞计数"vs"血常规"vs"全血细胞分析"这种**语义高度相似但本质不同**的术语;0.91 区间最易误判,必须有 LLM 兜底。

**为什么不用全量 LLM**:7 本 × 200 节点 ≈ 1400 节点,两两组合 ~100 万次,无论时间还是 API 成本都不可接受。

### 1.4 子问题 D:教学连贯性保障
**问题**:压缩可能切断知识链路(A 是 B 的前置,A 被删了 B 就讲不通)。
**我们的保障机制**:
1. **prerequisite 关系优先保护**:压缩时先看每个节点的入度/出度;高度数节点(被多个节点依赖)永远不会进 remove 候选
2. **merge 决策不可逆**:压缩比超额时只从 keep 决策中删,merge 永远保留(merge 已经是有意识的多源精华融合)
3. **整合后的 master_graph 包含 4 类关系**(prerequisite/parallel/contains/applies_to),前端可视化让人类用户也能审核知识链

### 1.5 子问题 E:压缩比的精确计算
**问题**:如何定义"原始字数"和"整合后字数"?

**约定**(写入代码 `core/integrator/pipeline.py`):
- **原始字数** = `sum(textbook.total_chars)`,即所有解析后正文字数
- **整合后字数** = `sum(node.definition + node.name + node.chapter)`,即 master_graph 的"知识压缩内容"(用户看到的全部信息)
- **压缩比** = 整合后字数 / 原始字数(目标 ≤ 30%)

注意:整合后的"内容"不是教材原文重组,而是结构化知识点的定义集合 + 图谱关系。这是赛题要求的"精华版本",不是"删减版"。

### 1.6 子问题 F:RAG 分块策略选择依据

**待选方案对比**(详见 `report/整合报告.md` 实测数据):

| 分块大小 | 每本 chunk 数 | top-5 命中率 | 平均响应时间 | 选用? |
|---|---|---|---|---|
| 300 字 + 50 重叠 | ~1500 | 0.71 | 1.8s | ❌ 单块信息不足 |
| **600 字 + 80 重叠** | ~750 | **0.86** | 2.1s | **✅ 选用** |
| 800 字 + 100 重叠 | ~600 | 0.79 | 2.4s | ❌ 相关度被稀释 |

**选 600 字的核心理由**:
- BGE-small-zh 上下文 512 token,中文 600 字 ≈ 380-450 token,落在窗口中央(向 query 端偏移 70%,即每块**前 60% 的信息密度被嵌入向量重点表征**)
- 一个完整概念的"定义+举例+应用"段落通常在 400-600 字
- 重叠 80 字 ≈ 一个完整句子,够防止概念切断

## 2. 输入输出边界

### 2.1 系统输入
1. **教材文件**(用户上传):PDF / MD / TXT / DOCX,任意数量任意大小(单文件验证至 437MB)
2. **用户问题**(自由文本):自然语言问题,中文为主
3. **教师反馈**(自由文本):自然语言修改建议

### 2.2 系统输出
1. **单本知识图谱**(每本一份 JSON,前端 ECharts 渲染)
2. **整合后主图谱**(去重 + 合并后,字数 ≤ 原始 30%)
3. **整合决策列表**(每条带 reason + confidence,可被对话修改)
4. **RAG 回答**(必带引用 [教材, 章节, 页码, 相关度])
5. **整合报告**(Markdown,包含决策摘要、压缩比、教学完整性分析)

## 3. 非功能需求

### 3.1 性能
- 7 本教材完整解析 + 抽取 + 整合 + 索引,**端到端 < 25 分钟**(实测见整合报告)
- 单次 RAG 查询 < 4 秒(含 LLM 生成)
- 前端 SPA 首屏 < 2 秒,图谱渲染(2000 节点)< 3 秒

### 3.2 可移植性
- 后端纯 Python 3.12,前端 Vite 构建静态产物
- 部署只需:Python 进程 + 一个静态目录 + ChromaDB 持久化文件夹
- 一键 Docker(`docker-compose up`)

### 3.3 可观测性
- Token 消耗实时统计(`core/agent/llm_client.py:TokenStats`)
- 解析/抽取/索引进度通过 `/api/textbooks` `/api/rag/status` 查询
- 所有 LLM 调用 latency 进 loguru 日志

### 3.4 容错
- 单章 LLM 抽取失败不阻塞整本(`kg_extractor.py` 有 try/except 隔离)
- LLM JSON 模式解析失败时,启用"截取首 `{` 到末 `}`"的兜底
- 节点 ID 全局唯一,防止跨章/跨书 ID 碰撞污染图谱
